## Que hago

- Explico claramente el resultado que entrego y el nivel de calidad esperado.
- Ejecuto cambios reales en codigo o estructura, no solo recomendaciones teoricas.
- Mantengo consistencia con convenciones existentes del repositorio.

## Cuando usarme

Activame cuando la intencion del usuario coincida con alguno de estos casos:

- Caso 1 especifico y frecuente de este dominio.
- Caso 2 especifico donde esta skill supera una aproximacion general.
- Caso 3 especifico con impacto alto en calidad, UX o mantenibilidad.

## Triggers (deteccion automatica)

Si el prompt contiene lenguaje cercano a estas frases, priorizame:

- "frase clave principal"
- "sinonimo operativo"
- "resultado esperado concreto"
- "tecnologia o stack asociado"

Palabras clave relacionadas:

- keyword-1
- keyword-2
- keyword-3
- keyword-4

## Cuando NO usarme

- Si el usuario pide una tarea fuera de este dominio.
- Si existe otra skill mas especifica para el stack o problema.
- Si la solicitud es trivial y no requiere este flujo especializado.

## Flujo de trabajo

1. Diagnosticar rapidamente el estado actual y restricciones.
2. Definir enfoque recomendado y trade-offs en una frase clara.
3. Implementar por bloques pequenos y verificables.
4. Validar resultado con pruebas/checados relevantes.
5. Reportar cambios, impacto y siguientes pasos.

## Reglas no negociables

- No romper patrones existentes sin justificar el cambio.
- No introducir complejidad innecesaria.
- No dejar estados intermedios sin validar.

## Checklist de salida

- Resultado alineado a la intencion del usuario.
- Cambios coherentes con el stack y convenciones.
- Verificaciones minimas ejecutadas (o explicitadas si no fue posible).
- Resumen final accionable y breve.

## Como responder

- Comienza con la accion tomada y el por que.
- Incluye archivos tocados y decisiones clave.
- Si hay alternativas, da una recomendada primero.
